# 17 — Bases Program (Milestone N): the vault as a database a screen-reader user can actually query

**Status:** 📝 Specs drafted (2026-07-06); implementation not started. GH [milestone 14](https://github.com/coryj627/slate/milestone/14), issues [#690–#711, #713](https://github.com/coryj627/slate/milestone/14) (N0: #690–#693, #713 · N1: #694–#698 · N2: #699–#701 · N3: #702–#706 · N4: #707–#711). Evidence base: [`01_research_brief.md`](01_research_brief.md) — the official Obsidian Bases help corpus (schema, expression language, complete function inventory), the two owner-pinned feature-request threads (transient quick search — shipped by Obsidian in 1.12.x; tasks in Bases — declined by Obsidian on cache-architecture grounds), and six field reports. Architecture authority: `../05_locked_architecture_decisions.md` §8 (queries, Bases, accessible data grid — decided 2026-05-17) and the [milestone 14 description](https://github.com/coryj627/slate/milestone/14) (normative subsections vendored: [`02_milestone_brief.md`](02_milestone_brief.md)), which this program refines into executable specs — **every deliberate divergence from either is recorded in [`specs/gap_analysis.md`](specs/gap_analysis.md)**; the program supersedes both where they differ (`../06_v1_milestones.md` rule).

**Strategic goal.** Ship **Bases v1**: `.base` files open in Slate and round-trip byte-safely with Obsidian; an accessible structural query builder produces the same query AST that `.base` parsing produces; results render in an accessible data grid (table + list) with sort, groupBy, summaries, transient quick filtering, in-grid property editing, and CSV/Markdown export; queries embed in notes and compose into saved queries and dashboards. Two capabilities go **beyond** Obsidian: (1) the query builder and data grid are designed for AT users first (05 §8.6–§8.7 — Obsidian's visual builder and canvas-adjacent grids are the capability reference, never the interaction reference), and (2) **tasks are queryable** — Slate's index already stores full task rows (migration 008), so `source: tasks` and `file.tasks.*` aggregates ship in v1 where Obsidian's metadata cache architecturally cannot (brief §5.2). The biggest V1.x milestone; sequenced after K–M so tester feedback shaped the function set and defaults (milestone 14 rationale).

Everything here inherits the UI-parity Presentation-Ready DoD (`../08_ui_parity/00_program.md` §A–§G): a11y-check 100/100 gated on each PR's own tip, APCA Lc ≥ 75 measured in both appearances, census-gated invariants, one PR per issue, fmt/clippy pre-push. This document adds only what is Bases-specific.

## Locked scope decisions (owner brief = milestone 14 description + 05 §8; refined 2026-07-06)

| # | Area | Decision |
|---|------|----------|
| 1 | Identity | **Milestone N — Bases v1.** Specs in `docs/plans/17_bases/`; phase prefixes N0–N4. Write scope is exactly three paths: (a) `.base` file save through the N0-3 serializer, (b) property edits routed through the **existing** frontmatter write path (`session.set_property`, session.rs:1713), (c) saved-queries/dashboards rows in SQLite. **Query execution itself is read-only** (census-gated, DoD §N-F). |
| 2 | Syntaxes | **Three syntaxes, one engine** (05 §8.1), **all three in N** (owner decision 2026-07-06, reversing this program's draft deferral: DQL is "necessary to support this work correctly"): `.base` YAML (first-class, round-trippable), the builder (produces the AST directly), and Dataview DQL — **parsed, not authored**: ` ```dataview ` block queries convert to the same AST and render read-only (N0-5 #713); Slate never writes DQL; conversion losses fail loud per decision 6, never approximate silently. **DataviewJS: never** (05 §8.1, no JS runtime) — ` ```dataviewjs ` fences stay ordinary code blocks. Inline `= expr` DQL is the reserved N-E1 remainder. |
| 3 | Round-trip | Hard requirement (05 §8.1, milestone DoD): open in Slate → save untouched → **byte-equal**. Targeted edits produce **minimal diffs** (key order, quoting, comments preserved where the YAML layer allows). Mechanism: preservation-first parse — raw YAML retained; unknown keys, unknown per-view state (brief §6.1), unknown functions, and unsupported view types are **preserved verbatim, never dropped, never reformatted**. Slate-authored view state lives under a `slate` sub-key inside the view entry so it can never collide with Obsidian's undocumented keys. Golden corpus of Obsidian-authored fixtures proves this in CI forever (§N-A). |
| 4 | View types | `table` + `list` **execute** in v1 (05 §8.5). `cards` / `map` / plugin types **parse-preserve** and render as an honest **table-fallback with a notice banner** naming the requested layout — never a refusal, never a blank surface. Cards is V1.x; Map is V2+ with the 05 §8.5 AT caveat (structured list, not synthesized map description). ` ```base ` / ` ```slate-query ` / ` ```dataview ` fences are discovered at scan time into a **new `bases_blocks` table**; `.base` files index into a new `bases_files` table (both in migration 021 — the 05 §4.5 sketch's `specialized_blocks` table was never implemented; reading-pipeline block kinds are derived at render time, so fence discovery brings its own table). |
| 5 | Expression language | **One language for filters and formulas** (brief §2), Obsidian-semantics where the brief documents them, with a **pinned v1 function table** (brief §3 minus `random()` — determinism §N-B; `html()`/`image()`/`icon()` parse but render as text in v1, recorded per function in the n1 spec) plus the degree extension fields `file.inDegree`/`file.outDegree` (links-index counts — the milestone-14 "ship the basics" commitment; full graph-derived fields stay N-E5). **Where JS coercion semantics are undocumented, Slate is deliberately more defined and less magical**: comparisons and arithmetic are total over data (Null/mixed types yield falsey/Null + a view warning, exactly as pinned in n1 §N1-1 rules 2–3), and only *structural* problems are errors (decision 6) — the delta from full-JS coercion is recorded in gap O13. The brief-§3 inventory is normative; additions are individually-tracked issues. |
| 6 | Fail-loud semantics | An expression the v1 evaluator **structurally** can't run — unknown function, `Unsupported` construct, wrong function arity/argument kind, `this` without a context, filter-only function misused — **fails the enclosing view loudly**: error banner naming the construct, no rows shown for that view, other views unaffected. **Never** silently include/exclude rows, never coerce a structural error to false. **Data-shaped conditions are not errors**: Null or cross-type comparisons evaluate falsey with a non-fatal view warning (n1 §N1-1 rules 2–3) — otherwise `price > 2.1` would kill every view on a vault where one file lacks `price`, since the default dataset is the whole vault (brief §1). Parse stays tolerant (unknown constructs become typed `Unsupported` nodes and round-trip); the structural/data boundary is the load-bearing line. (DoD §N-D.) |
| 7 | Engine & result shape | SQLite-backed with a Rust formula layer (05 §8.3): pushdown for indexed predicates (folder, tag incl. nested via shipped `SearchScope::Tag` semantics, extension, mtime/ctime, property equality via `value_text_norm` + `properties_list_values`, task columns), Rust-side evaluation for the rest, over a **paged candidate stream** (no load-everything, 05 §9.3.1). Results use the **rich tabular `BasesResultSet`** per 05 §8.4 (columns with `ColumnRole`, rows with `audio_description`, groups, summaries, `audio_summary`). The shipped FTS `QueryResultSet` (search_db.rs:100) is **not retrofitted** — it stays the FTS shape; composition happens via decision 9. Query cache keyed by (AST hash, vault generation), invalidated on source-set change; **cache ≡ fresh is census-gated** (§N-C). |
| 8 | Tasks | **First-class, v1, a deliberate differentiator** (owner-pinned request, brief §5.2): (a) Slate view extension `source: tasks` — rows are task items with a `task.*` namespace (text, status, completed, due, scheduled, priority, file); (b) `file.tasks.total` / `file.tasks.completed` aggregates usable in any filter/formula/column. Extension keys are chosen to be Obsidian-ignorable and are documented as Slate extensions with an interop caveat (a base using them shows an error in Obsidian's view but the file is untouched). |
| 9 | Full-text composition | Slate extension `file.matches("query")` in filter position routes to the existing FTS5 index and ANDs with structured predicates (05 §8.8). No snippet/score surfacing in v1 (columns come from properties). Same interop-caveat treatment as decision 8. |
| 10 | Editing | In-grid cell editing writes through the **existing** property write path (`set_property` / `delete_property`, session.rs:1713/:1755) — atomic, already tested, no new write machinery. Formula columns and immutable file properties are read-only with an AX hint. After an edit the view re-executes (cache invalidation makes this correct by construction). |

## Definition of Done — N-specific deltas

*Census convention, for readers new to the repo: a `census_*` test proves an invariant by generating large randomized inputs (plus exhaustive small-case sweeps) and checking the invariant over every one — see the existing `census_*` fns in `crates/slate-core/src/canvas` and `session/tests` for the shape. `census_scale()` is a per-module local helper (three signatures exist; copy the nearest one, there is no shared crate helper), and `SLATE_CENSUS_FULL=1` raises the scale for release runs. "Golden" tests are hand-written expected values (canvas convention) — no snapshot crate is in the workspace.*

- **§N-A (round-trip census):** golden Obsidian-authored corpus: parse → save untouched ⇒ **byte-equal**; parse → targeted edit → save ⇒ minimal diff (untouched lines byte-identical). Adversarial YAML census: random mutations (unknown keys at every level, weird quoting, comments, plugin view types, unknown functions) never panic, never drop content on re-save, always parse to *something* preservable.
- **§N-B (determinism):** same vault + same query ⇒ identical `BasesResultSet` including row order (explicit sort keys + path tiebreak); permutation of file-insertion order ⇒ identical results. No `random()`, no wall-clock in evaluation (`now()`/`today()` evaluate against an execution timestamp captured once per run and carried in the result for testability).
- **§N-C (cache ≡ fresh):** census — any interleaving of edits/renames/deletes with cached query re-runs yields results identical to a cold engine run.
- **§N-D (fail-loud):** census over the corpus × mutation set — no unsupported construct ever silently changes a result set; it either round-trips inertly (not referenced by the executed view) or fails that view with a named error.
- **§N-E (budgets):** decision-16 numbers recorded in `BENCHMARKS.md` at wave close; scan-bench diff proves N0-4 stays inside the first-open indexing budget.
- **§N-F (read-only query path):** census — executing every corpus query over a fixture vault leaves every vault byte identical (hash before/after). Writes happen only via the three decision-1 paths.
- **§N-G (projection & authoring equivalence):** table and list expose the same rows/data/actions per view; builder → AST → `.base` → AST round-trips to the same query; every builder-expressible query renders identically when re-opened from its saved `.base`.
